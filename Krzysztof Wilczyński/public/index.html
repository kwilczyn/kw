Let’s Take Care of Your Git Repository Together

For the past few years, I have worked as a Tech Lead responsible for, among other things, the quality of code automating tests at the Integration and System levels. Maintaining a dedicated repository (Gitlab) in good condition over several years required adherence to good practices related to the Git version control system by all team members. My responsibility was to maintain the highest standards and to train less experienced developers.

When it comes to order, I am a great advocate of using team tools such as Commitzen to help unify the style of committed messages while adhering to the Conventional Commits specification.

The commit message format compliant with the Conventional Commits specification forces developers to commit changes in a readable way, facilitating easy rollbacks (if needed) and allowing convenient tracking of project history as it evolves.

In my opinion, maintaining order in the code repository is invaluable. Collaboration with less experienced users can complicate the work of the entire team, which is why I have always offered constructive feedback and support.

Pull Requests have always been an ideal tool for giving constructive feedback for me. As a Lead, I was responsible for setting up the repository in such a way that no change could be added to the main branch without creating a mandatory Pull Request and receiving Approval from another team member. As with commit messages, I am also an advocate of standardization in the form of Conventional Comments, which improves clarity of communication and facilitates sharing constructive feedback.

Conclusion: I am a candidate with a particular commitment to good practices related to maintaining a code repository. I can’t wait to make a positive contribution to maintaining your repository as well.

Testing Pays Off

As an experienced tester, I had the opportunity to be involved in implementing functional tests at all levels, starting from unit tests (jest, pytest, vitest), through integration tests (pytest) and ending with E2E Tests (RobotFramework, Selenium, Playwright, Cypress).

From experience, I know that technical debt in the form of lack of test coverage at lower levels of the test pyramid will sooner or later cause large amounts of additional work for the entire team (and thus increase the cost of the entire project). In such circumstances, new manual testers are often hired for the project, or attempts are made to cover the application with additional automated tests at the E2E level using a dedicated team of specialists. Unfortunately, E2E tests are expensive and very sensitive tests.

How much cheaper and more stable would it be to ensure test coverage before writing the first line of code (TDD), or simply maintain test coverage regularly.

As a person who previously worked in a testing position, I know techniques for designing good tests that I will be able to use in practice as a developer as well. I can easily prepare unit tests, integration tests, and component tests for my new code. Covering acceptance criteria with E2E level tests in any technology will not be a problem for me either. Concepts of tools such as stubs and mocks are not unfamiliar to me, so lack of readiness of external APIs will never stand in the way of covering the frontend part with tests.

Conclusion: My experience in software testing will be used to better cover code with tests at the unit level and beyond.
